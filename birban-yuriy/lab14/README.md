## Шаблони

- DRY (Don’t Repeat Yourself) – уникати дублювання коду.
- KISS (Keep It Simple, Stupid) – простота важливіша за складність.
- YAGNI (You Aren’t Gonna Need It) – не реалізовувати функціонал заздалегідь.
- SOLID – набір із п’яти принципів ООП:

S — Single Responsibility Principle (Принцип єдиної відповідальності)
Клас повинен виконувати лише одну задачу і мати одну причину для змін. Наприклад, клас User не має відповідати за збереження у базу чи відправку пошти.

O — Open/Closed Principle (Принцип відкритості/закритості)
Клас має бути відкритий для розширення, але закритий для модифікації. Нову поведінку додаємо через нові класи-нащадки, а не змінюємо існуючий код.

L — Liskov Substitution Principle (Принцип підстановки Лісков)
Об’єкти підкласів повинні замінювати базові класи без порушення роботи програми. Наприклад, функція, що працює з Bird, має коректно працювати з Duck, що наслідує Bird.

I — Interface Segregation Principle (Принцип розділення інтерфейсу)
Клієнти не повинні залежати від методів, які вони не використовують. Краще створити кілька вузьких інтерфейсів, ніж один великий.

D — Dependency Inversion Principle (Принцип інверсії залежностей)
Високорівневі модулі не повинні залежати від низькорівневих; обидва повинні залежати від абстракцій. Наприклад, клас має залежати від інтерфейсу IDatabase, а не від конкретного класу MySQLConnection.
